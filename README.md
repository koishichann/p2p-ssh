# p2p-ssh
scripts using for establish p2p-ssh connection on linux.

P2P SSH 隧道
项目介绍
本工具是一套简单的纯手工P2P SSH隧道方案，无需依赖第三方工具，无需购买云服务器，仅通过Linux原生工具socat实现，用于解决两台NAT内网下Linux设备的SSH/SFTP连接问题，适合个人自用。
核心特点：零成本、操作简单、无第三方依赖，自动清理残留进程，不浪费网络资源，支持SSH命令行和SFTP文件传输。
- ✅ 无第三方依赖：不使用Tailscale、FRP，不依赖任何云服务器/第三方服务
- ✅ Agent友好：服务端脚本无阻塞，执行后立即退出，Agent可正常捕获输出并回复
- ✅ 智能打洞：最大发送60次UDP打洞包，连接建立后立即停止发包，不浪费流量
- ✅ 自动清理：5分钟无连接自动关闭隧道，无进程残留
- ✅ 全系统适配：支持Ubuntu、Debian、CentOS、WSL openEuler（自动适配apt/yum包管理器）
- ✅ 完整功能：支持SSH命令行控制、SFTP文件传输，连接稳定不掉线
- ✅ 安全加密：基于socat原生传输加密，结合SSH自身加密，双重防护，流量点对点直连
原理说明（通俗版）
核心逻辑：利用「NAT打洞」技术，绕过NAT内网「禁止外部主动连接」的限制，通过手动传递双方网络信息，实现两台内网Linux设备点对点直连，全程无中转。
核心原理步骤
1. 终端A（发起连接方）启动脚本：自动安装socat、生成随机端口、获取自身公网IP，前台阻塞运行，等待终端B信息。
2. 手动传递信息：将终端A的公网IP+端口，传递给终端B的使用者。
3. 终端B（被连接方）启动脚本：自动安装socat、生成随机端口、获取自身公网IP，后台运行，发送最多60次UDP打洞包（用于打通NAT通道）。
4. 连接检测：终端B检测到隧道建立（有SSH流量），立即停止发送打洞包；若5分钟无连接，自动关闭隧道。
5. 建立连接：终端A输入终端B的公网IP+端口，隧道打通，通过本地端口SSH连接终端B。
关键补充
- 打洞包：仅用于打通NAT通道，连接建立后立即停止，避免无限发包占用资源。
- 隧道保活：SSH连接建立后，SSH自带心跳包持续产生流量，维持隧道稳定，不会触发超时关闭。
- 端口说明：隧道端口（随机生成）与SSH业务端口（22）完全独立，隧道仅负责传输，不修改、不转发端口。
环境要求
- 系统要求：两台设备均需为Linux系统（支持apt/yum包管理器）
- 权限要求：必须使用root权限运行脚本（安装socat、启动隧道需root）
- 网络要求：两台设备均需联网（用于安装socat、获取公网IP）
- 前置条件：被连接方（终端B）需开启sshd服务（默认开启，若未开启需手动安装openssh-server）、可在命令行执行命令、运行有已接入消息平台且能收到用户即时消息的agent
使用步骤（全程复制命令即可，个人自用简化版）
第一步：准备工作（两台设备均执行）
1. 下载脚本：将两个脚本（客户端、服务端）分别下载到两台Linux设备，放在任意目录（如 /root 目录）。
客户端脚本：p2p-ssh-client.sh
服务端脚本：p2p-ssh-server.sh
2. 赋予脚本执行权限（两台设备均执行）：
chmod +x p2p-ssh-client.sh p2p-ssh-server.sh
第二步：启动客户端（终端A，发起连接方，前台阻塞）
1. 运行客户端脚本（root权限）：
./p2p-ssh-client.sh
2. 脚本运行后，会输出客户端公网信息（示例）：
========= 客户端信息 =========
你的公网地址: 183.xx.xx.xx:54321
本地SSH端口: 2222
==============================
请让 Agent 执行：./p2p-ssh-server.sh 183.xx.xx.xx:54321
请输入服务端返回的 IP:PORT → 
3. 保持该窗口打开（阻塞运行，关闭窗口会断开隧道），将输出的「公网地址」手动传递给终端B使用者。
第三步：启动服务端（终端B，被连接方，后台运行）
1. 在终端B中，执行以下命令（替换为终端A的公网IP:端口）：
./p2p-ssh-server.sh 183.xx.xx.xx:54321
2. 脚本执行后立即退出（无阻塞），输出服务端信息（示例）：
========= 服务端运行成功 ==========
你的公网地址: 113.xx.xx.xx:56789
自动清理: 5分钟无连接自动关闭隧道
===================================

✅ P2P 隧道连接已建立成功！
📶 对端客户端信息：183.xx.xx.xx:54321
打洞包已发送: 18次（最大发送60次）
3. 将输出的「服务端公网地址」手动返回给终端A使用者。
第四步：建立SSH连接（终端A）
1. 在客户端阻塞窗口中，输入Agent返回的服务端IP:端口（示例）：
请输入服务端返回的 IP:PORT → 113.xx.xx.xx:56789
2. 隧道建立成功，脚本提示：
✅ 隧道建立中，保持本窗口打开
✅ 连接命令：ssh root@127.0.0.1 -p 2222
3. 新建一个命令行窗口，执行连接命令，即可远程登录家里WSL：
ssh root@127.0.0.1 -p 2222
4. SFTP使用：打开WinSCP/FileZilla，协议选择SFTP，主机127.0.0.1，端口2222，用户名root，密码为家里WSL root密码，即可上传下载文件。
自定义参数（服务端脚本）
服务端脚本顶部可自定义以下参数（按需修改）：
# ========== 自定义参数 ==========
MAX_PUNCH_COUNT=60        # 最大打洞包次数（默认60次）
PUNCH_INTERVAL=1          # 打洞包发送间隔（默认1秒/次）
SOCAT_TIMEOUT=300         # 无流量自动关闭时间（默认300秒=5分钟）
# =================================
常见问题
1. 脚本运行报错「socat: E exactly 2 addresses required」
原因：socat参数语法错误，逗号后面加了空格（如 UDP-LISTEN:56789, fork 是错误的）。
解决：确保脚本中socat参数逗号后无空格（如 UDP-LISTEN:56789,fork），重新运行脚本。
2. 客户端输入服务端IP:端口后，无法建立连接
原因1：服务端打洞包已发送完毕（60次），未及时建立连接，隧道已超时关闭。
解决：重新让Agent执行服务端脚本，立即在客户端输入服务端IP:端口。
原因2：家里WSL未开启sshd服务。
解决：在WSL中执行 systemctl start sshd，并设置开机自启 systemctl enable sshd。
3. 服务端进程残留，无法重新启动
解决：手动清理残留进程：
kill $(cat /tmp/p2p-socat.pid) 2>/dev/null
4. SSH连接后，长时间不操作会断开
原因：SSH心跳包未正常触发（极少数情况）。
解决：无需处理，本脚本已默认适配SSH心跳，连接建立后不会因无操作断开；若仍断开，可在客户端SSH连接时添加心跳参数：ssh root@127.0.0.1 -p 2222 -o ServerAliveInterval=30。
注意事项
- 必须使用root权限运行脚本，否则无法安装socat、启动隧道。
- 客户端脚本必须保持前台阻塞运行，关闭窗口会导致隧道断开。
- 服务端脚本执行后立即退出，Agent可正常返回结果，无需等待。
- 连接建立后，打洞包会立即停止发送，仅SSH心跳包维持隧道，不浪费流量。
- 若家里网络重启，公网IP会变化，需重新执行脚本、传递新的公网信息。
- 本方案仅用于个人远程运维，请勿用于未授权的网络访问。
总结
本工具是个人自用的NAT内网穿透方案，零成本、无第三方依赖，用于解决两台NAT下Linux设备的远程连接问题，操作简单，支持SSH和SFTP，适合个人日常远程运维、文件传输使用。
